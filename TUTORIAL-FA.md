# راهنمای کامل استفاده از QA Test Generator

> تمام دستورات به صورت **اینتراکتیو** (با پرامپت) و **CLI** (با فلگ) قابل استفاده هستند.
> این راهنما روی حالت **اینتراکتیو** تمرکز دارد.

---

## شروع کار (یک بار انجام بده)

### ۱. تنظیم LLM Provider

```bash
qa config
```

سوالات رو جواب بده:
- **Add / edit a provider** → انتخاب کن
- **Provider** رو انتخاب کن (مثلاً Ollama برای لوکال، یا Gemini/OpenRouter برای کلود)
- **Model** رو وارد کن
- **Active provider** رو تأیید کن

### ۲. بررسی اتصال

```bash
qa models
```

اگر لیست مدل‌ها نمایش داده شد، اتصال برقراره.

## دستورات و نحوه ترکیب آنها

### نقشه کلی

```
qa new                    ─── شروع: ساخت پروژه
qa generate-guide         ─── اختیاری: یادگیری از پروژه موجود
qa scenario               ─── اختیاری: نوشتن سناریو
qa hybrid / qa analyze    ─── اصلی: تولید تست از صفحه وب
qa generate               ─── تکمیلی: تولید تست تکی
qa fix                    ─── تعمیر: رفع خطای تست
qa autonomous             ─── پیشرفته: خزش خودکار سایت
qa chat                   ─── کمکی: چت با AI
qa docs                   ─── مستندات: تولید داکیومنت
```

## مرحله ۱: ساخت پروژه

```bash
qa new
```

سوالات اینتراکتیو:
1. **نام پروژه** → مثلاً `my-cypress-tests`
2. **مسیر** → پیش‌فرض `./my-cypress-tests`
3. **زبان** → TypeScript (پیشنهاد) یا JavaScript
4. **BDD** → آیا Cucumber BDD میخوای؟ (معمولاً Yes)
5. **Allure** → آیا Allure Reporter میخوای؟ (معمولاً Yes)
6. **Base URL** → آدرس اپلیکیشن مورد تست
7. **npm install** → اجرا بشه؟ (Yes)
8. **LLM-Wiki** → راهنمای AI بسازه؟ (Yes — مفیده)
9. **Sample scenarios** → نمونه سناریو بذاره؟ (Yes)

**خروجی:** یه پروژه کامل Cypress با ساختار POM + BDD + Allure

---

## مرحله ۲: تولید تست (۳ روش اصلی)

### روش A: `qa hybrid` — بهترین دقت (توصیه شده)

ترکیب Playwright + AI: اول صفحه رو بازدید میکنه، المنت‌ها رو استخراج میکنه، بعد AI تست مینویسه.

```bash
qa hybrid
```

سوالات:
1. **URL صفحه** → مثلاً `https://example.com/login`
2. **نام** → مثلاً `LoginPage`
3. **احراز هویت** → آیا صفحه لاگین میخواد؟
4. **حالت تعاملی** → آیا مرورگر باز بشه برای تعامل دستی؟
5. **تایر تست** → smoke یا regression

**خروجی:** ۳ فایل تولید میشه:
- `cypress/e2e/locators/LoginPageLocators.ts` — انتخابگرها
- `cypress/e2e/pages/loginPagePage.ts` — شیء صفحه
- `cypress/e2e/test/smoke/loginPage.cy.ts` — تست

---

### روش B: `qa analyze` — تحلیل صفحه

مشابه hybrid ولی فقط تحلیل Playwright (بدون AI پیشرفته):

```bash
qa analyze
```

سوالات:
1. **URL صفحه**
2. **احراز هویت** (اختیاری)
3. **نام** (اختیاری — از عنوان صفحه استخراج میشه)
4. **تایر تست**
5. **فایل سناریو** → آیا سناریو داری؟
6. **ذخیره سناریو** → سناریو تولید شده رو ذخیره کن؟

**نکته:** اگه `--scenario-output scenarios/login.md` بذاری، سناریو ذخیره میشه و بعداً میتونی باهاش کار کنی.

---

### روش C: `qa generate` — تولید با AI

تولید تکی یا گروهی فایل‌ها با AI:

```bash
qa generate all
```

سوالات:
1. **توصیف** → مثلاً "صفحه لاگین با نام کاربری و رمز عبور"
2. **URL** → آدرس صفحه
3. **نام** (اختیاری)
4. **منبع سناریو:**
   - **AI خودش بنویسه** (Phase 0) — پیشنهاد میشه
   - **خودم تایپ کنم**
   - **از فایل بخونه**
5. **تایر تست**

## ترکیب دستورات برای بهترین نتیجه

### گردش کار ۱: ساده (سریع)

```
1. qa new                    ─── ساخت پروژه
2. qa hybrid                 ─── تولید تست برای هر صفحه
3. npm run cy:smoke:all      ─── اجرای تست‌ها
```

### گردش کار ۲: حرفه‌ای (با سناریو)

```
1. qa new                    ─── ساخت پروژه
2. qa scenario               ─── نوشتن سناریو با AI
   → سناریو ذخیره میشه در scenarios/login.md
3. qa g all --scenario-file scenarios/login.md --name LoginPage
   ─── تولید تست از سناریو
4. npm run cy:smoke:all      ─── اجرا
```

### گردش کار ۳: تحلیل + اصلاح

```
1. qa new                    ─── ساخت پروژه
2. qa analyze                ─── تحلیل صفحه + ذخیره سناریو
   → سناریو ذخیره میشه
3. سناریو رو دستی ویرایش کن   ─── بهترش کن
4. qa g all --scenario-file scenarios/edited.md --name Page
   ─── تست نهایی رو بساز
5. npm run cy:smoke:all      ─── اجرا
```

### گردش کار ۴: پیشرفته (مرحله به مرحله)

```
1. qa new --llm-wiki         ─── ساخت پروژه با راهنمای AI
2. qa scenario               ─── سناریو بنویس
3. qa g locators -g "..."    ─── فقط انتخابگرها رو بساز
4. qa g page -g "..."        ─── فقط شیء صفحه رو بساز
5. qa g test -g "..."        ─── فقط تست رو بساز
6. npm run cy:smoke:all      ─── اجرا
```

### گردش کار ۵: خزش خودکار سایت

```
1. qa new                    ─── ساخت پروژه
2. qa auto                   ─── خزش خودکار سایت
   → برای هر صفحه تست تولید میشه
3. npm run cy:smoke:all      ─── اجرا
```

## توضیح هر دستور

### `qa scenario` — نوشتن سناریو

سناریو یه فایل Markdown هست که مراحل تست رو توصیف میکنه.

```bash
qa scenario
```

سوالات:
1. **توصیف** → مثلاً "ورود به سایت با نام کاربری و رمز عبور"
2. **حلقه ویرایش** (تا ۱۰ بار):
   - **ذخیره** → فایل ذخیره بشه
   - **اصلاح** → چی تغییر کنه؟
   - **بازسازی** → دوباره از اول
   - **لغو**
3. **نام فایل** → مثلاً `login`

**خروجی:** `scenarios/login.md`

**نمونه سناریو:**
```markdown
# Scenario: Login

1. **Visit** /login
2. **Type** "admin" into **username input**
3. **Type** "password123" into **password input**
4. **Click** **login button**
5. **Assert** URL contains "/dashboard"
```

---

### `qa generate-guide` — یادگیری از پروژه موجود

```bash
qa gg
```

اگه پروژه Cypress دیگه‌ای داری، این دستور ساختارش رو یاد میگیره و یه فایل راهنما میسازه. بعداً توی دستورات دیگه با `--guide` استفاده میشه.

---

### `qa fix` — تعمیر تست خراب

```bash
qa fix
```

سوالات:
1. **مسیر فایل تست** → مثلاً `cypress/e2e/test/smoke/login.cy.ts`
2. **پیام خطا** → متن خطایی که دیدی
3. **Stack trace** (اختیاری)
4. **اعمال تعمیر؟**

AI خطا رو تحلیل میکنه و نسخه تعمیر شده رو پیشنهاد میده.

---

### `qa chat` — چت با AI

```bash
qa chat
```

یه جلسه چت تعاملی با AI. میتونی سؤال بپرسی، کد بخوای، یا کمک بخوای.

دستورات ویژه:
- `/reset` — پاک کردن تاریخچه
- `/help` — نمایش کمک
- `/exit` — خروج

---

### `qa autonomous` — خزش خودکار

```bash
qa auto
```

سایت رو میخزه و برای هر صفحه‌ای که پیدا کنه تست تولید میکنه.

سوالات:
1. **آدرس پایه** → مثلاً `http://localhost:3000`
2. **عمق خزش** → چند لینک دنبال بشه؟ (۱-۳)

---

### `qa steps` — تولید فایل مراحل

```bash
qa steps
```

یه فایل JSON میسازه که مراحل تعامل با صفحه رو توصیف میکنه. بعداً با `qa hybrid --steps-file` استفاده میشه.

```bash
# ابتدا مراحل رو بساز
qa steps -g "روی دکمه کلیک کن، فرم رو پر کن، ارسال کن"

# بعد با hybrid استفاده کن
qa hybrid --url http://localhost:3000 --steps-file steps/steps.json
```

## نکات مهم

### همیشه اول `qa config` رو اجرا کن
بدون تنظیم LLM، هیچ دستور AI کار نمیکنه.

### از `--guide` استفاده کن
اگه پروژه‌ات ساختار خاصی داره، با `qa gg` یه راهنما بساز و توی دستورات دیگه با `--guide` استفاده کن. این باعث میشه AI کد رو مطابق ساختار پروژه‌ات بنویسه.

### سناریو = کلید کیفیت
هرچی سناریو دقیق‌تر باشه، تست بهتر تولید میشه. از `qa scenario` برای نوشتن سناریو استفاده کن، بعد دستی ویرایشش کن، بعد با `qa g all --scenario-file` تست بساز.

### Interactive vs CLI
- **Interactive:** فقط `qa hybrid` بدون فلگ بزن — سوالات پرسیده میشه
- **CLI:** فلگ‌ها رو بذار — `qa hybrid -u URL -n Name -y`

### تست‌ها رو اجرا کن
```bash
cd my-cypress-tests
npm run cy:smoke:all     ─── اجرای تست‌های smoke
npm run cy:bdd:all       ─── اجرای تست‌های BDD
```
